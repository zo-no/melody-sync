## 当前已经完成什么

### 1. 前端和 workbench 结构已经明显收口

- `static/chat/` 已经按 `core / session / session-list / workbench / settings` 分域。
- `workbench-ui.js` 不再承担全部细节，地图、tracker、branch actions、operation record 已经拆分。
- hooks 设置和 node 设置已经进入统一设置弹窗。

### 2. hooks 现在已经有明确生命周期契约

- 生命周期事件已经固定为 `instance / session / run / branch` 这套闭环。
- 设置面板不再按技术字段解释，而是按用户感知的流程展示。
- `builtin.branch-candidates` 不只是写状态事件，现在也会生成 hook 来源的 `taskMapPlan`。

### 3. node 已经不是单纯的前端硬编码枚举

当前 node 结构已经拆成 6 层：

1. `durable truth`
   - `session / run / taskCard / branchContext / continuity`
2. `node kind catalog`
   - `chat/workbench/node-definitions.mjs`
3. `composition contract`
   - root 能力、父子 kind 约束、默认交互、默认边类型、默认 view type、surfaceBindings、capabilities
4. `node effects`
   - `static/chat/workbench/node-effects.js`
5. `default continuity projection`
   - `static/chat/workbench/task-map-model.js`
6. `taskMapPlan overlay`
   - `chat/workbench/task-map-plans.mjs`
   - `static/chat/workbench/task-map-plan.js`

### 4. 右侧无限画布已经改成 `view.type` 驱动

现在 node / plan 可以声明右侧展示方式：

- `flow-node`
- `markdown`
- `html`
- `iframe`

这意味着后面如果 AI 判断某个内容更适合 HTML、Markdown 或嵌入页展示，不需要改 renderer，只需要改 node / plan 数据。

### 5. `augment-default` 已经能覆盖默认节点

这次补上的关键能力是：

- `taskMapPlan` 不再只能追加新节点
- 当 plan 使用和默认 continuity 投影相同的 node id 时，会合并节点元数据
- 现在 hook / AI 可以给默认节点补：
  - `summary`
  - `capabilities`
  - `surfaceBindings`
  - `view`

这一步很关键，因为它让“默认 continuity 地图”和“hook / AI 产出的图谱意图”真正接上了。

## 当前真实架构应该怎么理解

### 1. 核心真值仍然是 session-first

现在不要误解成“地图已经变成 workflow 真值”。还没有。

当前真值仍然是：

- `Session`
- `Run`
- `taskCard`
- `branchContext`
- `continuity`

地图、右侧画布、建议卡片，本质上都还是表达层。

### 2. hook 可以写图谱表达层，但不能直接夺走 durable truth

现在更准确的关系是：

```text
durable state
  -> 默认 continuity 投影
  -> 默认 task map

lifecycle hook
  -> 可选 taskMapPlan overlay
  -> 给默认地图补节点元数据 / 富视图 / surface 绑定

renderer
  -> 统一负责布局、DOM、iframe、安全渲染
```

### 3. branch 能力已经开始和 node 汇合，但还没有彻底统一

现在支线能力和 node 的关系是：

- branch 真值仍在 `taskCard + branchContext + session`
- `candidate / branch / done` 已经能在地图里表达这条生命周期
- `builtin.branch-candidates` 已经可以把支线建议转成 hook 生成的 map overlay

但还没有彻底统一到“同一个 node instance 同时驱动地图节点和底部建议卡”。

## 当前还没有做完的关键点

### 1. 地图候选节点还是双来源

现在 `candidate` 仍然有两条来源：

- 默认 continuity 投影会根据 `taskCard.candidateBranches` 生成候选节点
- `builtin.branch-candidates` hook 也会生成同 id 的 overlay 节点，用来补 metadata 和后续富视图

这一步已经能工作，但还不够干净。下一阶段需要继续把候选节点收成单一表达层。

### 2. 底部建议卡还没有正式切到 node / plan 驱动

你之前明确提出过：支线出现时有两件事同时发生：

1. 会话底部出现卡片
2. 地图里出现对应支线 node

现在地图这边已经有 plan contract 了，但底部建议卡还主要走旧的 `taskCard.candidateBranches` 读取路径，没有完全切到 node / plan。

### 3. 还没有真正的 `node instance store`

现在我们已经有：

- `node kind`
- `composition contract`
- `node effects`
- `taskMapPlan`

但还没有独立的 `GraphNode / NodeInstance` 持久层。也就是说，当前更像“图谱计划层”，还不是“图谱实例层”。

### 4. AI 还没有正式接入 plan 生成链

现在 contract 已经足够让 AI 读懂：

- 有哪些 node kind
- 每个 node kind 能挂在哪
- 默认交互是什么
- 允许哪些 edge type
- 允许哪些 `view.type`

但 AI 还没有正式进入“读取 contract -> 生成 `taskMapPlan` -> 渲染”的生产链。

## 下一阶段准备做什么

下面这几步是我建议的顺序，优先级已经排好。

### 第一阶段：把 branch suggestion 真正统一成一个表达层

目标：

- 一个候选支线只维护一份表达
- 同时投影到：
  - 地图 node
  - 底部建议卡
  - 后续可能的 tracker 提示区

建议动作：

1. 引入 `surface projection` 概念
2. 让 `candidate` node 的 `surfaceBindings` 真正驱动 UI
3. 底部建议卡优先从 plan / node surface 读取，而不是单独读 `taskCard.candidateBranches`

### 第二阶段：补 `node instance + capability` 层

目标：

- node 不再只是“地图上的块”
- node 成为真正的图谱工作单元

建议最小结构：

- `GraphNode`
  - `id`
  - `kind`
  - `status`
  - `sourceSessionId`
  - `sessionId`
  - `branchContextId`
  - `surfaceBindings`
  - `view`
  - `capabilities`

- `GraphEdge`
  - `structural`
  - `suggestion`
  - `completion`
  - `merge`

### 第三阶段：把 AI 接到 plan 生成链

目标：

- AI 不直接改 renderer
- AI 不直接给坐标
- AI 不直接写 DOM / CSS
- AI 只生成 `taskMapPlan`

需要补的东西：

1. 给 AI 的最小输入上下文
   - 当前 session 摘要
   - 当前 branch 状态
   - node kind contract
   - edge / capability / surface / view contract
2. plan 校验层
3. hook / AI plan 优先级策略

### 第四阶段：把更多右侧内容收成 type-driven canvas

你的目标是“右侧无限画布可以展示各种用户数据”。这条线已经起头了，后续建议继续往下面扩：

1. `html`
2. `markdown`
3. `iframe`
4. 流程图 / graph / 结构化文档块
5. 文件型视图

关键原则不变：

- AI 改的是 plan / node / recipe 文件
- 底层 renderer 保持稳定

### 第五阶段：继续清老结构

当前还需要继续清理的重点：

- `chat/workbench-store.mjs` 还偏大
- 老的 workbench project/node/capture 结构和现在的 task-map node 仍然并存
- settings/general 这条线刚开始露出，还没完全并入架构说明

后续继续开刀时，优先做“边界收口”，不要反向再做大一统文件。

## 后续开发的工作原则

### 1. 不要再把能力直接写死在 renderer

以后如果要加：

- `goal` node
- `review` node
- `decision` node
- `html panel` node

优先改：

- node kind definition
- composition contract
- hook recipe / plan producer
- taskMapPlan

最后才考虑 renderer 是否真的需要新能力。

### 2. 不要让 hook 变成 workflow 真值

hook 可以：

- 写事件
- 写 plan
- 写外部通知
- 写 surface projection

hook 不应该直接接管：

- session durable state
- run durable state
- branchContext 真值

### 3. UI 尽量改声明，不改底层

你已经明确希望“页面的大部分 UI 内容由 AI 控制，并且通过改文件而不是改底层代码实现”。后面就按这个原则推进：

- 优先改 contract / recipe / plan 文件
- 其次改 projection
- 最后才动 renderer

## 当前这轮改动的直接结论

一句话总结：

当前 MelodySync 已经从“session continuity + 前端硬编码地图”进入到了“session-first durable truth + hook-generated plan overlay + type-driven right canvas”的阶段。

这还不是最终形态，但下一阶段要做的方向已经很清楚了：

- 继续把 branch 能力收进 node
- 继续把 UI surface 收成 plan / projection 驱动
- 继续把 AI 的控制入口固定在 contract + plan，而不是 renderer
