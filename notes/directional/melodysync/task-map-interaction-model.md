# MelodySync｜主线任务地图交互模型

> 文档定位：这份文档定义下一版 owner 主交互模型。它不描述当前已上线结构，而是约束下一步产品迭代应该如何把 `主线任务`、`支线任务`、`聊天工作区` 分开。

## 1. 要解决的问题

当前体验的根本问题不是“列表不够好看”，而是前台把三种对象混在了一起：

- 主线任务
- 支线任务
- 聊天会话

这会导致两个直接后果：

- 用户不知道自己到底在做哪件事
- 用户必须读很多信息，才能理解当前上下文

下一版的目标只保留两个：

1. 用户始终知道自己正在推进哪条主线、哪个节点。
2. 用户不需要读大量聊天或列表摘要，也能知道现在该做什么。

## 2. 核心产品对象

下一版前台应该固定成三层：

### 2.1 主线任务 `MainQuest`

这是左栏列表里的对象。

左栏以后只放主线任务，不放支线任务，不放 branch session。

每个主线任务至少表达：

- 名称
- 当前激活节点
- 总体进度
- 支线摘要

### 2.2 任务节点 `TaskNode`

这是中栏任务地图里的对象。

它统一承载：

- 主线路径节点
- 正式支线节点
- 候选支线节点
- 说明、决策、参考资料这类富文本节点

节点才是“当前工作的焦点单位”。

节点不应该被限制成“只有任务”。

只要能帮助用户更快理解当前结构，AI 可以直接创建：

- markdown 笔记节点
- 决策节点
- 参考资料节点
- 阶段总结节点

### 2.3 当前节点工作区 `Workspace`

这是右栏对象。

右栏只服务当前激活节点，不直接服务整棵任务树。

`当前任务 Bar` 负责回答：

- 我现在在哪个节点
- 它属于哪条主线
- 它的父节点是谁
- 下一步要做什么

聊天区负责推进当前节点，不负责承载任务结构。

## 3. 左中右三栏职责

### 左栏：主线任务列表

左栏只做“任务入口”和“总览恢复”：

- 我有哪些主线任务
- 当前在做哪一条
- 哪条停住了
- 哪条已经完成较多

它不再负责：

- 展示完整支线树
- 展示 branch session 列表
- 展示大量状态细节

### 中栏：任务地图

中栏是当前主线任务的树状流程图。

它是任务结构视图，不是消息视图，不是列表摘要视图。

用户应该在这里一眼看到：

- 当前主线从哪里开始
- 当前走到了哪一步
- 哪些支线已经出现
- 哪些支线正在推进
- 哪些支线已经完成或挂起

### 右栏：当前节点工作区

右栏是当前激活节点的工作台。

结构固定为上下关系：

- 上：当前任务 Bar
- 下：聊天与执行区

## 4. 中栏任务地图应该长什么样

中栏不建议做成普通树列表，建议做成“主线脊柱 + 侧挂支线”。

也就是：

- 主线节点沿一条主路径排布
- 支线从某个主线节点旁边长出来
- 当前路径高亮
- 当前节点最强高亮
- 已完成支线折叠
- 候选支线用虚线或弱化节点表达

这套心智更接近游戏地图：

- 主线是主路
- 支线是岔路
- 当前节点是当前位置

而不是：

- 左侧有一堆会话项
- 中间再有一堆任务项
- 右边还有一堆聊天项

## 5. 支线不应该怎么出现

支线不应该按“检测到对话偏离主话题”就立即创建。

这是系统内部判断，不是用户心智。

如果只按偏离检测创建支线，用户会觉得：

- 为什么突然多了一条任务
- 这条任务是建议，还是已经开始了
- 它和主线是什么关系

## 6. 支线应该怎么出现

支线应该先作为“地图上的可选分叉”出现，再决定是否变成正式节点。

推荐把支线分成三种展示状态：

### 6.1 候选支线

AI 判断这里值得单独推进，但还没有真正创建。

中栏表现：

- 挂在当前节点旁边
- 虚线节点
- 轻量标题
- 一行原因或目标

用户动作只有三种：

- 进入处理
- 稍后再说
- 忽略

### 6.2 正式支线

当用户点“进入处理”，或者系统已经必须独立推进时，候选支线变成正式支线。

这时：

- 节点状态变成 `进行中`
- 右栏工作区切到这个支线节点
- 当前路径同步高亮

### 6.3 已回收支线

支线完成后不应该消失。

它应该：

- 在地图上保留为已完成或已带回主线节点
- 默认折叠
- 把结论写回父节点

## 7. 什么情况下才值得生成支线

生成支线的判断标准应该是“是否值得独立推进”，不是“是否偏题”。

满足下面条件再生成候选支线：

- 这个子问题有独立目标
- 它有可回收的结果
- 它预计会持续多轮
- 它适合暂停和恢复
- 它做完后能并回父节点

否则就继续在当前节点里处理，不生成支线。

## 8. 一整套推荐交互流程

### 8.1 创建主线任务

用户输入一个新的工作目标。

例如：

- 做一个电影史学习计划
- 把这个 PR 的 CI 问题彻底修掉

### 8.2 系统先解析目标

不要直接让用户掉进长聊天。

先生成一份轻量骨架：

- 主目标
- 完成标准
- 足够表达结构的节点骨架
- 默认起始节点

这里不应该再给 AI 限制成“只先给接下来的三个”。

如果任务本身已经明显存在多层结构，AI 可以一次性生成更深的任务图。

### 8.3 中栏立刻出现任务地图

用户先看结构，再开始推进。

### 8.4 右栏激活当前节点

`当前任务 Bar` 应明确显示：

- 当前节点
- 主线名称
- 父节点
- 下一步

### 8.5 聊天推进当前节点

聊天的默认对象不再是“整个任务”，而是“当前节点”。

### 8.6 出现值得拆开的子问题

系统先在中栏生成候选支线节点，而不是直接开出一个新的会话表项。

### 8.7 用户选择是否进入支线

如果进入：

- 中栏高亮这条支线
- 右栏切到该支线节点

如果稍后：

- 节点保留为挂起候选

如果忽略：

- 节点消失或折叠为轻量提示

### 8.8 支线完成并回收

支线完成后：

- 节点状态改为已完成或已带回主线
- 父节点得到简短回流结论
- 焦点回到父节点或主线路径下一步

## 9. 降低信息负担的显示规则

为了让用户不用读太多内容，中栏和右栏应遵守这些硬规则：

- 一次只允许一个当前激活节点
- 当前路径永远高亮
- 已完成支线默认折叠
- 地图节点只显示标题、状态、一句话摘要
- 地图默认只展开当前路径和当前节点附近一层
- 左栏只显示主线任务，不再显示支线

## 10. 下一版最小范围

第一版不应该做太大，建议只做这些：

- 左栏只保留主线任务列表
- 中栏实现树状任务地图
- 右栏实现“当前任务 Bar + 当前节点工作区”
- 支线以候选节点形式出现
- 点击节点可以切换当前工作焦点
- 支线完成后可折叠回父节点

同时增加两个约束：

- 任务图创建权默认交给 AI，不要求用户手工约束整张图的深度和节点种类
- 文案默认追求绝对精简，例如 `开启`、`已开启`，并避免无意义的 `空闲`

先不要做这些：

- 自由编辑复杂图结构
- 无限深度自动分叉
- 地图里塞入大量文本
- 把每个节点都暴露成完整会话列表

## 11. 对当前代码的约束

这一版前台语义已经不适合继续把 `session` 暴露成主对象。

但底层实现可以先不推翻，短期建议是：

- `session` 继续作为执行容器
- `task map node` 成为前台主对象
- 主线与支线关系优先通过节点层表达
- 旧的 session/branch 逻辑先映射到节点，而不是继续直接暴露

也就是说：

```text
session = runtime container
task node = product object
main quest = navigation object
```

而不是：

```text
session = user-facing object
branch session = user-facing side task
```

## 12. 一句话结论

下一版不该继续优化“会话列表怎么展示支线”，而应该切成：

- 左栏看主线
- 中栏看地图
- 右栏做当前节点

支线首先是地图上的分叉，不是会话列表里的新项。
