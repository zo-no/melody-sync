# MelodySync PRD｜工作连续性 MVP

> 文档定位：这是 MelodySync 第一版可执行产品定义，不再讨论宏大叙事，只定义第一版到底做什么、为什么做、如何验收。

## 1. 一句话定义

MelodySync 是一个面向长期认知工作的 AI 协作工作台，用来解决人和 AI 在分叉、切换、长期推进过程中的上下文断裂问题，并把有效解法逐步沉淀成可复用能力。

## 2. 这个版本要解决的核心问题

当前真正要解决的问题不是“AI 与人类如何共生”，而是更具体的工作连续性问题：

- 用户在聊天过程中经常岔开到相关分支，回来后丢失原工作位置。
- 长尾项目推进时，人和 AI 都会反复失忆，不知道之前已经讨论到哪里。
- 一次会话中得到的有效解法难以泛化，无法在下次对话或相似任务中直接复用。
- 普通聊天记录只能保存过程，不能稳定转化为可维护的项目结构和下一步动作。

## 3. 目标用户与第一场景

第一目标用户只有一个：你自己。

第一核心场景只有一个：产品研发工作台。

原因很简单：这个场景同时天然包含随机输入、分支讨论、长期项目推进、任务拆解、总结沉淀和方法复用，是验证 MelodySync 是否成立的最佳切口。

## 4. 产品目标

本版产品只验证一件事：

持续聊天，能否稳定沉淀为一个可维护、可推进、可回溯的项目结构。

如果这件事不成立，后续的技能树、社区共建、语音优先、AI 社会叙事都不成立。

## 5. MVP 不做什么

本版明确不做：

- 社区共建
- 点赞筛选优质 skill
- 复杂技能树画像
- 自动化程度过高的智能推荐
- 完整语音工作流
- 多项目复杂调度
- 面向大众的通用人生管理

## 6. MVP 核心闭环

第一版必须完整跑通下面这条闭环：

聊天 -> 提取节点 -> 进入 Inbox -> 归类到 Project -> 挂到 Tree -> 从节点继续聊 -> 阶段总结 -> 写回 Obsidian

任何超出这条闭环的功能，都应该延后。

## 7. MVP 核心对象

### 7.1 会话记录 Session

原始过程层，记录说过什么、做过什么。它重要，但不是长期资产本身。

### 7.2 收集项 Capture Item

从会话中提取出的候选内容，先进入 Inbox，等待整理。

建议最小字段：
- id
- sourceSessionId
- sourceMessageSeq
- text
- kind
- status
- createdAt

### 7.3 项目 Project

长期推进容器，例如一个具体产品或一个明确主题。

建议最小字段：
- id
- title
- brief
- obsidianPath
- status
- rootNodeId

### 7.4 节点 Node

项目树中的正式元素。节点类型先控制在：问题、观点、方案、任务、风险、结论、知识。

建议最小字段：
- id
- projectId
- parentId
- title
- type
- summary
- sourceCaptureIds
- state
- nextAction
- createdAt

### 7.5 分支上下文 Branch Context

这是保证“岔开不丢线”的关键对象。记录分支从哪里开始、解决什么、回来接到哪里。

建议最小字段：
- id
- nodeId
- sessionId
- goal
- returnToNodeId
- checkpointSummary
- createdAt

### 7.6 Skill（仅做雏形）

不是每次会话都能变成 skill。只有被验证过、可复用、可迁移的方法，才允许升级为 skill。

建议最小字段：
- id
- title
- trigger
- procedure
- evidenceNodeIds
- status

## 8. MVP 核心界面

本版不重做整体框架，沿用当前 MelodySync 聊天壳。

界面结构：

- 左侧：Session List
- 中间：Chat
- 右侧：Work Panel

右侧 Work Panel 第一版只做两个 Tab：

- Inbox
- Tree

Skill 不做独立主视图，先作为节点的后续提升结果存在。

## 9. MVP 核心动作

### 动作 1：提取到 Inbox

用户从当前聊天内容中，手动把一段话、一条消息、一个结论提取到 Inbox。

第一版先手动，不强求 AI 自动识别。

### 动作 2：归类到 Project

用户把 Capture Item 放入某个 Project，并指定它要成为哪类 Node。

### 动作 3：挂到 Tree

一旦归类，Capture Item 就转为正式 Node，挂入项目结构树。

### 动作 4：从节点继续聊

用户点击某个 Node，从该 Node 的上下文继续开一个分支会话，而不是从零开始聊天。

### 动作 5：阶段总结

系统围绕当前 Project、某个 Branch 或某个 Node 输出阶段性总结，包括：
- 当前结论
- 待办
- 未决问题
- 下一步动作

### 动作 6：写回 Obsidian

把项目树、阶段总结、关键节点沉淀回明确的 Obsidian 文件，而不是只停留在产品内部。

## 10. 人和 AI 的职责边界

人负责：
- 提出真实问题
- 做关键判断与取舍
- 补充隐性上下文
- 决定哪些内容值得长期沉淀

AI 负责：
- 帮助维护上下文连续性
- 记录主线与分支关系
- 提取候选节点
- 组织项目树
- 生成阶段总结草稿
- 识别哪些路径可能具备 skill 化价值

一句话：
人类负责学习、判断、提供关键信息；AI 负责记忆、组织、提示、复用。

## 11. 为什么它比 Obsidian / 普通聊天更强

Obsidian 强在存储，但弱在过程连续性。它能记下来，但不能天然维护“我为什么岔开、我现在该回哪”。

普通聊天强在即时互动，但弱在结构稳定性。它能聊，但会把问题埋进时间流里。

MelodySync 如果成立，它的优势只有一个：

它不是帮用户保存内容，而是帮用户保持工作连续性，并把连续工作里的有效路径沉淀成可复用能力。

## 12. 第一版实现顺序

### Phase 1：手动闭环跑通
- Chat 保存
- 手动提取 Capture Item 到 Inbox
- 手动归类到 Project Tree
- 手动触发阶段总结
- 写回 Obsidian

### Phase 2：半自动整理
- AI 自动提取候选节点
- AI 建议节点类型
- AI 建议归属 Project
- AI 生成总结草稿

### Phase 3：Skill 雏形
- 从被验证过的节点路径中提取 Skill
- 支持在新对话中提示可调用 Skill

## 13. MVP 验收标准

满足以下条件，就认为 MVP 成立：

- 你能连续用它推进一个真实产品项目至少一周。
- 你在中途岔开话题后，能在 30 秒内回到原工作位置。
- 聊天内容不会只停留在会话历史里，而能进入项目结构。
- 至少有一个被验证过的方法，能从节点路径沉淀成可复用 skill 雏形。

## 14. 当前开放问题

这些问题不影响 PRD 成立，但会影响后续实现细节：

1. Inbox 是全局唯一，还是每个 Project 各自有局部 Inbox？
2. 分支会话是否直接复用现有 Session 模型，还是引入更强的 Branch 实体？
3. 写回 Obsidian 时，是一个 Project 对应一个主文档，还是拆成 Tree / Summary / Skill 多文档结构？
4. Skill 第一版是独立对象，还是 Node 的一种特殊状态提升？

## 15. 本文档结论

MelodySync 第一版不是知识库，不是聊天工具，不是任务软件，而是一个用于维护长期工作连续性的 AI 协作系统。

它的北极星不是“记录更多”，而是：

让人和 AI 在长期工作中，不再反复失去彼此已经共同获得的上下文、位置和能力。
