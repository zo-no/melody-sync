# MelodySync PRD｜工作连续性 MVP

> 文档定位：这是 MelodySync 第一版可执行产品定义。它不讨论宏大叙事，只定义第一版到底做什么、为什么做、如何验收。

## 1. 一句话定义

MelodySync 是一个让人和 AI 围绕目标任务树持续推进长期工作的协作系统。它通过任务分叉、回溯点和阶段产物，解决聊天导致的上下文断裂，并把被验证过的做法逐步沉淀为可复用能力。

## 2. 这个版本真正要解决的问题

当前真正的问题不是“如何做一个更大的知识库”，而是更具体的工作连续性问题：

- 用户在聊天过程中经常岔开到相关分支，回来后丢失原工作位置。
- 长尾项目推进时，人和 AI 都会反复失忆，不知道当前目标推进到哪里。
- 一次会话中得到的有效解法难以泛化，无法在下次对话或相似任务中直接复用。
- 普通聊天记录只能保存过程，不能稳定转化为可推进的目标路径。

## 3. 产品判断

MelodySync 第一版的主结构不再定义为“节点树”，而定义为“目标任务树”。

原因如下：

- 你的核心痛点不是“缺少知识节点”，而是“岔开之后不知道如何回来”。
- 节点树更像知识组织结构，任务树更像工作推进结构。
- 第一版必须先解决“当前在做什么、为什么分叉、回来接哪里”这三个问题。

因此，MelodySync 第一版不是知识图谱产品，也不是通用待办工具，而是一个围绕目标推进的连续性系统。

## 4. 目标用户与第一场景

第一目标用户只有一个：你自己。

第一核心场景只有一个：产品研发工作台。

原因很简单：这个场景天然同时包含长期项目推进、话题分叉、回到主线、阶段总结和方法复用，是最适合验证工作连续性的切口。

## 5. 产品目标

本版产品只验证一件事：

持续聊天，能否稳定沉淀为一个可维护、可推进、可回溯的目标任务路径。

如果这件事不成立，后续的技能树、社区共建、语音优先、AI 社会叙事都不成立。

## 6. 已固定的 MVP 规则

下面这些规则在第一版直接写死，不再作为开放讨论项：

1. 任务创建采用 `AI 建议 + 用户确认`，不允许完全自动落树。
2. 任务分叉默认创建子任务，不支持同级旁支作为默认分叉方式。
3. `checkpoint` 和 `summary` 是两个独立对象，不混成一个中间对象。
4. Obsidian 写回采用多文档结构，不把所有内容塞进一个大文件。
5. 一个会话在任意时刻只绑定一个当前任务，不允许同时挂多个任务。

## 7. MVP 不做什么

本版明确不做：

- 社区共建
- 点赞筛选优质 skill
- 复杂技能树画像
- 自动化程度过高的智能推荐
- 完整语音工作流
- 多项目复杂调度
- 面向大众的通用人生管理
- 通用知识图谱式节点系统
- 自动多任务并行挂载
- 一个会话同时服务多个任务

## 8. MVP 核心闭环

第一版必须完整跑通下面这条闭环：

聊天 -> AI 提炼当前目标 -> 用户确认当前任务 -> 在当前任务下继续聊 -> 目标变化时分叉新子任务 -> 生成 checkpoint / summary -> 写回 Obsidian -> 从已验证路径里提炼 skill 候选

任何超出这条闭环的功能，都应该延后。

## 9. MVP 核心对象

### 9.1 会话记录 Session

原始过程层，记录说过什么、做过什么。每个会话都服务于一个且仅一个当前任务。

建议最小字段：
- id
- currentTaskId
- title
- status
- createdAt
- updatedAt

### 9.2 项目 Project

长期推进容器，例如一个具体产品或一个明确主题。一个 Project 下维护一棵目标任务树。

建议最小字段：
- id
- title
- brief
- obsidianPath
- status
- rootTaskId

### 9.3 目标任务 Task

第一版的主结构单位。它不是普通待办，而是一个有明确目标、状态、上下文和下一步动作的推进单元。

建议最小字段：
- id
- projectId
- parentTaskId
- title
- goal
- status
- branchReason
- currentContext
- nextAction
- latestCheckpointId
- createdAt
- updatedAt

建议状态先控制在：
- open
- active
- done
- parked

### 9.4 回溯点 Checkpoint

这是保证“岔开不丢线”的关键对象。它记录某个任务在某个阶段到底推进到了哪里，方便中断后迅速恢复。

建议最小字段：
- id
- taskId
- summary
- decisions
- unresolved
- nextAction
- createdAt

### 9.5 阶段总结 Summary

围绕一个任务、一个分支或整个项目生成的阶段性总结，用来做收敛、回看和写回。

建议最小字段：
- id
- projectId
- taskId
- scope
- content
- createdAt

### 9.6 Skill（仅做雏形）

不是每次会话都能变成 skill。只有被验证过、可复用、可迁移的方法，才允许升级为 skill。

建议最小字段：
- id
- title
- trigger
- procedure
- evidenceTaskIds
- status

## 10. 任务树与产物的关系

第一版要明确区分“主结构”和“产物”：

- 主结构是任务树。
- 会话是过程。
- 总结、结论、决策、方法是任务产物。
- skill 是被验证过的方法产物，不直接充当树节点。

这条边界必须守住。否则产品会重新退化成“所有东西都想挂树上”的知识堆积器。

## 11. 任务层级与分叉规则

第一版只支持单父级任务树。

### 11.1 何时创建新任务

满足任一条件时，AI 可以提出创建新任务的建议：

- 当前聊天已经不再服务于原目标。
- 用户明确说“另开一个相关问题”。
- 现有任务的下一步动作已经变成另一个独立目标。

### 11.2 分叉默认规则

第一版默认把分叉创建为“子任务”。

理由：

- 子任务天然表达“我是从哪个目标里岔出来的”。
- 子任务更利于恢复路径。
- 第一版先不要引入同级旁支的复杂判断。

### 11.3 何时更新当前任务而不是新建

如果聊天仍然在为同一个目标补充上下文、澄清方案、推进下一步，就只更新当前任务，不新建任务。

### 11.4 何时关闭任务

满足以下条件之一时，可以把任务标记为 `done` 或 `parked`：

- 目标已完成，进入 `done`。
- 当前不再推进，但未来可能恢复，进入 `parked`。
- 当前任务已经被更具体的子任务替代，父任务保留为容器节点。

### 11.5 第一版不做合并

任务合并在概念上合理，但第一版不做自动合并，也不把人工合并放进主流程。避免树结构早期变得不可控。

## 12. 会话绑定规则与切换机制

第一版必须遵守下面这条硬规则：

一个会话在任意时刻只绑定一个当前任务。

这条规则的含义是：

- 当前会话里的所有消息都默认服务于这个任务。
- 如果目标发生变化，不是把同一个会话同时挂到多个任务上，而是先创建新子任务，再把会话切换到新的当前任务。
- 会话可以从任务 A 切到任务 B，但切换发生时必须留下 checkpoint。

推荐切换机制：

1. AI 检测到目标变化，提出“建议创建子任务”。
2. 用户确认后，系统创建子任务并记录 branch reason。
3. 原任务立刻生成一个 checkpoint，保存到切换前状态。
4. 当前会话的 `currentTaskId` 切到新任务。

## 13. 用户最小操作流

### 13.1 开始工作

用户进入某个项目，创建或确认一个当前任务，然后开始聊天。

### 13.2 发生分叉

聊天中出现新目标时，AI 提示“这已经是一个新任务”，用户确认后创建子任务并切换过去。

### 13.3 暂离当前任务

如果用户要离开当前任务，系统先生成 checkpoint，再允许切走。

### 13.4 恢复工作

用户回到某个任务时，系统先展示最近 checkpoint，再把会话继续绑定到这个任务。

### 13.5 收尾沉淀

当某个任务阶段完成时，生成 summary，并把任务树、checkpoint、summary、skill 候选写回 Obsidian。

## 14. MVP 核心界面

本版不重做整体框架，沿用当前 MelodySync 聊天壳。

界面结构：

- 左侧：任务列表（桌面侧栏 / 移动端抽屉）
- 中间：Chat
- 顶部：任务 bar（轻量焦点条）

第一版不再单独做右侧 Task Panel，而是把任务相关信息拆成两层：

- 左侧任务列表负责完整任务树、生命周期状态和恢复入口
- 顶部任务 bar 只负责当前焦点，不承担完整任务管理
- 桌面端通过 hover 暴露当前主任务下的子任务结构
- 移动端通过显式下拉按钮暴露同一套子任务结构
- 进入子任务时，顶部保留一条弱化主线引用，再展示当前子任务和轻量动作

如果后续需要任务候选区，也只能作为轻量 staging 区，不能喧宾夺主变成主结构。

## 15. MVP 核心动作

### 动作 1：AI 提炼当前目标

在聊天过程中，AI 基于整段上下文总结“当前真正要推进的目标任务”是什么。

第一版必须经过用户确认，不允许直接自动创建正式任务。

### 动作 2：生成或更新当前任务

如果当前聊天仍然服务于原目标，就更新当前任务的上下文、状态和下一步动作。

### 动作 3：目标变化时创建子任务

如果聊天目标发生变化，不再硬塞进原任务，而是在当前任务下创建一个新的子任务，并记录 branch reason。

### 动作 4：从任务继续聊

用户点击某个任务，从该任务的上下文继续开聊，而不是从零开始聊天。

### 动作 5：生成 checkpoint

系统围绕当前任务生成高频轻量的 checkpoint，用于恢复工作位置。

### 动作 6：生成 summary

系统围绕当前任务或项目生成低频收敛型的阶段总结。

### 动作 7：写回 Obsidian

把任务树、checkpoint、summary、技能候选沉淀回明确的 Obsidian 文件，而不是只停留在产品内部。

### 动作 8：从已完成任务中提炼 skill 候选

当一个任务路径被验证有效时，允许把它提炼成一个 skill 候选，供后续对话复用。

## 16. AI Hooks 设计原则

第一版允许存在 AI hooks，但只允许服务于连续性，不允许为了“看起来更智能”而增加复杂度。

允许的 hooks：

- 提炼当前任务
- 检测目标是否发生偏移
- 建议是否应该创建子任务
- 生成 checkpoint 草稿
- 生成阶段 summary 草稿
- 提示某条已验证路径可能值得升级为 skill

不允许的 hooks：

- 自动生成大量未确认任务
- 自动改写整个任务树
- 自动给出过度复杂的知识分类
- 自动推动用户进入与当前目标无关的新流程
- 自动把一个会话挂到多个任务

## 17. 人和 AI 的职责边界

人负责：
- 提出真实问题
- 做关键判断与取舍
- 决定当前什么目标最重要
- 决定是否确认 AI 建议的任务
- 决定哪些产物值得长期沉淀

AI 负责：
- 帮助维护任务连续性
- 识别当前目标
- 维护任务与子任务关系
- 生成 checkpoint / summary 草稿
- 提示哪些做法可能具备 skill 化价值

一句话：
人类负责判断优先级和价值，AI 负责记忆、组织、提示、复用。

## 18. 为什么它比 Obsidian / 普通聊天更强

Obsidian 强在存储，但弱在工作位置恢复。它能记下来，但不能天然告诉你“你原来正在推进什么、为什么岔开、现在应该接回哪里”。

普通聊天强在即时互动，但弱在连续推进。它能聊，但会把目标埋进时间流里。

MelodySync 如果成立，它的优势只有一个：

它不是帮用户保存更多内容，而是帮用户维持目标推进的连续性，并把有效路径沉淀成可复用能力。

## 19. Obsidian 写回结构

第一版直接采用多文档结构：

- `tasks.md`：记录任务树和任务状态
- `checkpoints.md`：记录任务级回溯点
- `skills.md`：记录 skill 候选和已确认 skill

如果后续需要 `summary.md`，可以增补，但第一版不把所有内容塞进同一个主文件。

## 20. 第一版实现顺序

### Phase 1：手动闭环跑通
- Chat 保存
- AI 建议当前任务 + 用户确认
- 手动或半手动创建子任务
- 手动触发 checkpoint / summary
- 写回 Obsidian

### Phase 2：半自动维护连续性
- AI 自动建议任务更新
- AI 自动建议是否分叉为子任务
- AI 生成 checkpoint 草稿
- AI 生成 summary 草稿

### Phase 3：Skill 雏形
- 从被验证过的任务路径中提取 skill 候选
- 支持在新对话中提示可调用 skill

## 21. MVP 验收标准

满足以下条件，就认为 MVP 成立：

- 你能连续用它推进一个真实产品项目至少一周。
- 你在中途岔开话题后，能在 30 秒内回到原工作位置。
- 系统始终能说明当前会话服务于哪个任务，以及它在整棵任务树中的位置。
- 会话切换任务时一定会留下 checkpoint，而不是静默丢失上下文。
- 至少有一个被验证过的方法，能从任务路径沉淀成可复用 skill 雏形。

## 22. 当前开放问题

第一版还剩下的真正决策已经很少，只剩这两项：

1. 第一次进入一个全新项目时，是先手动创建根任务，还是允许 AI 基于首轮对话建议根任务名称与目标？
2. `summary.md` 是否第一版就单独写出，还是先只保留 `tasks.md`、`checkpoints.md`、`skills.md` 三个文件？

## 23. 本文档结论

MelodySync 第一版不是知识图谱，不是聊天工具，也不是通用待办软件，而是一个用于维护长期工作连续性的目标任务协作系统。

它的北极星不是“记录更多”，而是：

让人和 AI 在长期工作中，不再反复失去彼此已经共同获得的上下文、位置和方法。
