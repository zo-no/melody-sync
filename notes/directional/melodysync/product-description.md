# MelodySync｜产品描述

> 文档定位：这是一份面向后续迭代的产品总述文档。它回答 MelodySync 现在是什么、先服务谁、解决什么问题、当前产品由哪些核心对象组成，以及下一轮迭代应围绕什么继续推进。

## 1. 一句话描述

MelodySync 是一个让人把模糊但反复出现的数字工作交给 AI、并在长期任务中保持工作连续性的跨端 AI 工作台。

它让用户可以从手机、桌面或其他入口提交任务，由运行在真实机器上的 AI worker 执行，并把聊天、执行、阶段产物和后续复用沉淀在同一条可恢复的工作线程里。

## 2. 当前产品定位

MelodySync 当前不是一个泛聊天工具，也不是一个移动 IDE。它的定位更接近：

- 一个运行在真实机器之上的 AI 自动化工作台
- 一个帮助用户先澄清问题、再组织执行的协作控制面
- 一个围绕长期任务推进、而不是一次性问答设计的持久化工作线程系统
- 一个把有效工作流逐步包装为 `App` 的复用层

当前阶段最重要的产品判断有两条：

- 第一，先解决真实可落地的重复数字工作，而不是先做宏大的知识系统。
- 第二，先解决长期协作中的工作连续性问题，而不是把产品做成更多会话、更重编辑或更多并发的驾驶舱。

## 3. 目标用户

### 3.1 当前核心用户

当前第一目标用户是产品的直接建设者本人，以及与其相似的重度脑力工作者。他们有几个共同特征：

- 日常工作里有大量反复出现的数字化事务
- 需要持续推进长期项目，而不是只要一次性答案
- 经常在聊天、任务、文档、执行之间来回切换
- 已经愿意和 AI 协作，但不想每次都从零重新讲清上下文

### 3.2 后续可扩展用户

当个人闭环成立后，MelodySync 才适合扩展到更广的一类人：

- 有重复数字工作、但没有自动化背景的普通用户
- 小团队里的 owner、管理者、运营或分析角色
- 希望把某个重复流程逐步沉淀为可复用入口的人

## 4. 要解决的核心问题

MelodySync 当前聚焦解决两类问题。

### 4.1 重复数字工作无法顺畅交给 AI

很多用户知道自己有一类重复工作值得自动化，但问题通常是模糊的：

- 输入材料不完整
- 输出目标不清楚
- 约束条件散落在聊天、截图、样例文件里
- 用户自己并不会直接写出高质量执行指令

因此，产品不能只做“把 prompt 发给模型”的转发层，而要承担问题澄清、方案组织和执行接力的职责。

### 4.2 长期任务推进中的工作连续性断裂

当任务持续数天到数周后，真正的痛点会变成：

- 聊天中一旦分叉，很难回到主线
- 人和 AI 都容易失去“当前推进到哪一步”的感觉
- 阶段结论、已做决策和待办动作混在消息流里
- 一次有效做法很难在下次任务中复用

MelodySync 因此需要把“持续聊天”升级成“可维护、可回溯、可复用的工作路径”。

## 5. 核心价值主张

如果产品成立，用户得到的不是一个更花哨的聊天框，而是三类更明确的价值：

- 更容易开始：用户不用先把问题想得特别完整，系统会协助澄清。
- 更容易推进：任务可以在不同设备之间切换，AI 在真实机器上完成执行。
- 更容易续上：中断、分叉、重启之后，仍能恢复到清晰的工作状态。

更进一步，MelodySync 希望把单次完成任务，逐步升级为沉淀 workflow、模板和 `App` 的长期能力积累。

## 6. 产品边界

### 6.1 MelodySync 是什么

- AI worker 的跨端控制面
- 长任务推进中的上下文恢复系统
- durable session 与执行结果的统一承载层
- workflow packaging 与复用入口

### 6.2 MelodySync 不是什么

- 终端模拟器
- 传统 editor-first IDE
- 默认面向 AI 专家的并发会话驾驶舱
- 单纯分享 prompt 的平台
- 通用多用户聊天 SaaS
- 试图替代 `codex`、`claude` 等强执行器本身的封闭运行时

这条边界的意义在于，MelodySync 不把主要精力投入到底层执行器竞争，而是围绕用户任务澄清、工作连续性和复用机制建立上层产品价值。

## 7. 当前产品语法

当前产品由四个公开对象和一条底层规则组成。

### 7.1 公开产品对象

- `Session`：持久化的工作线程，是用户感知的主对象
- `Run`：会话内的一次执行尝试
- `App`：启动会话时可复用的 workflow / policy package
- `Share snapshot`：不可变的只读会话导出

### 7.2 当前关键规则

- `Session` 是产品中心，`Run` 只是其下的一次执行
- 浏览器是控制面，不是系统事实来源
- HTTP 是规范状态路径，WebSocket 只做失效提示
- 持久状态必须落盘，运行进程可中断可重启
- 当前产品默认是单 owner 模式，visitor 通过 `App` 进入受限流程

## 8. 当前已形成的产品能力

基于仓库当前实现，MelodySync 已经具备以下基础能力：

- 用户可从手机或桌面访问同一套控制面
- 会话、事件和运行状态可持久化到磁盘
- 控制面重启后，长任务仍可通过持久状态恢复
- 可接入 `codex`、`claude` 等本地执行器，并保持 adapter-first
- 支持 `App` 作为流程入口与 visitor 范围控制
- 支持只读分享快照
- 支持自动生成会话标题、分组等展示元数据
- 前端保持无构建、轻量、跨端可访问

这意味着 MelodySync 已不再只是概念验证，而是已经具备“跨端发起 + 真机执行 + 持久线程 + 基础复用”的产品骨架。

## 9. 当前迭代主线

虽然产品已经有了通用控制面和执行骨架，但接下来不应继续平均发力，而要收敛到一条更清晰的主线：

把 MelodySync 从“可远程控制 AI worker 的系统”推进成“能够稳定承载长期目标任务树的工作连续性系统”。

这条主线意味着后续迭代重点不是继续堆更多零散能力，而是把下面几件事真正做顺：

- 当前到底在推进哪个目标
- 何时只是补充上下文，何时应该分叉成新任务
- 中断前应该留下什么 checkpoint
- 恢复时用户首先该看到什么
- 哪些阶段产物值得沉淀为 summary、决策或 skill 候选

## 10. 当前版本建议聚焦的 MVP

结合现有 PRD，下一阶段的 MVP 应定义为：

一个让人和 AI 围绕目标任务树持续推进长期工作的协作系统。

它的最小闭环应当是：

聊天 -> AI 提炼当前目标 -> 用户确认当前任务 -> 围绕该任务继续推进 -> 发生目标变化时分叉子任务 -> 生成 checkpoint / summary -> 写回外部知识系统或产品内沉淀 -> 从稳定路径中提炼 skill 候选

这个版本先只验证一件事：

持续聊天，是否真的能稳定沉淀为一个可维护、可推进、可回溯的目标路径。

## 11. 下一轮产品设计重点

为了让后续迭代不发散，下一轮产品设计应优先回答以下问题：

### 11.1 任务主结构

- 产品主结构是否明确以 `Project -> Task Tree -> Checkpoint / Summary` 为核心
- 一个会话在任意时刻是否只绑定一个当前任务
- 分叉默认是否总是形成子任务

### 11.2 连续性机制

- checkpoint 的最小字段是什么
- summary 与 checkpoint 如何分工
- 恢复工作时默认展示哪个对象

### 11.3 沉淀与复用

- 哪些阶段产物写回外部系统，哪些留在产品内
- skill 候选的提炼标准是什么
- 何时从一次任务升级为可复用 `App`

### 11.4 交互与界面

- 当前聊天壳如何承载任务面板、分叉建议和恢复入口
- 如何在不破坏轻量控制面的前提下增加任务结构可见性
- 手机端和桌面端各自最关键的信息密度应该是什么

## 12. 当前版本的成功标准

这一轮产品迭代不看宏大叙事，先看三个实际结果：

- 用户是否愿意持续把长期任务放在 MelodySync 里推进
- 一个真实的产品研发任务是否比之前更不容易断线
- 聊天内容是否开始稳定沉淀为可复用的结构资产，而不只是历史消息

如果这三点成立，后续再继续扩展技能树、社区共建、更多入口和更强的自动化才有意义。
