# README 概念对齐草案

这份草案只做一件事：先把 MelodySync 对外表述里的概念层级收紧，避免 README、产品文档、外宣稿各讲各的。

## 1. 先统一层级，不要把所有理念挤在第一屏

建议按四层来讲：

1. 对外主价值
   MelodySync 是一个让人把模糊但反复出现的数字工作交给 AI、并在长期任务中保持工作连续性的跨端工作台。
2. 产品机制
   用户从手机或桌面发起，AI 先帮他把问题讲明白，再让真实机器上的强执行器把事做掉，过程和上下文都能持续保留。
3. 工作连续性
   主支线、任务提醒、checkpoint、长期项、任务地图，都是为了不让长期任务在聊天中失焦。
4. 深层理念
   人和 AI 的共生、生产资料平权、同调、个人工作方式被产品反向塑形。

结论：README 首页主要讲 1 和 2，轻量带出 3，4 只保留一句到两句作为设计动机。不要一上来先讲宏大叙事。

## 2. 词汇建议

- `Session`
  继续作为当前 shipped product 的公开对象。它是运行时和架构都已经稳定的词。
- `Quest`
  适合作为更高一层的用户意义词，表示一条完整的目标生命线，而不是首页直接替换 `Session`。
- `Task`
  适合表示当前正在推进的具体目标。
- `Branch`
  适合表示从主目标中分出来的独立子问题。
- `长期项`
  对外更建议解释成“可复用资产”或“长期工作流”，不要单独丢一个黑话词。

结论：现阶段不要在 README 首页里把 `Quest` 当作底层对象名硬推翻 `Session`。更稳的做法是：
`Session` 讲系统对象，`Quest` 讲你真正想承载的用户体验。

## 3. README 应该先回答的三个问题

1. 这是什么产品？
   一个让普通人也能把重复数字工作交给 AI 的跨端工作台。
2. 它和普通 AI 聊天工具有什么不同？
   它不只给答案，而是帮你澄清问题、在真实机器上执行，并把长任务持续推进下去。
3. 它为什么长这样？
   因为真正的问题不是“怎么多开几个会话”，而是“怎么让人和 AI 一起完成一个长期目标，并且下次不用从头再来”。

## 5. 外宣短版发布稿 v1

### 可直接发布版

MelodySync 是一个让人把模糊但反复出现的数字工作交给 AI 的跨端工作台。你可以在手机或桌面上把任务、截图、样例文件直接交给它，AI 会先帮你把问题讲明白，再让真实机器上的执行器和本地工具把事情做掉。和普通 AI 聊天工具不同，它不会把一次对话当成一次性问答，而是把工作线程、执行状态和关键上下文持续保留下来，让任务在跨设备、跨天、被打断之后还能继续，而不是每次都重开重讲。

### 更短渠道版

MelodySync 不只是和 AI 聊天，而是一个让人把重复数字工作交给 AI 的跨端工作台。它会先帮你澄清任务，再在真实机器上执行，并把工作线程保留下来，让任务跨设备、跨天也能继续。

### 发布约束

- 默认不要在短版外宣里先讲 `Session`、`Quest`、`Branch`、`长期项`
- 第一段只讲三件事：服务谁、怎么做、为什么不是一次性聊天
- 默认不要在 headline 或首段点底层执行器品牌名，除非面向开发者渠道
- `Session` 适合放在后续产品机制说明里，不适合抢第一句 headline
