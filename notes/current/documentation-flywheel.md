# 文档管理与项目飞轮

状态：draft

目的：定义 MelodySync 当前阶段的文档管理方式，让文档不只是“记录”，而是能持续推动产品判断、重构执行、功能实现和回归校验，形成项目飞轮。

相关文档：

- `../../docs/README.md`
- `README.md`
- `core-domain-contract.md`
- `core-domain-refactor-todo.md`
- `session-run-closure-requirements.md`

## 一句话结论

文档不能按“写了多少”来管理，而要按“是否推动项目继续往前”来管理。

对当前项目来说，文档体系要形成下面这条飞轮：

```text
产品目标
  -> 领域边界
  -> 架构与代码拆分
  -> 功能实现
  -> 测试与验收
  -> 复盘与清理
  -> 回写文档
  -> 下一轮产品目标
```

如果某份文档不能落进这条链路，它就不该长期占据主线注意力。

## 为什么要这样管理

当前仓库里的文档不少，但问题不是“数量多”，而是容易出现三种失效方式：

1. 同一层内容有多份文档重复表达
2. 文档停留在讨论，没有回写到代码和测试
3. 新方向出来后，旧阶段文档还在主线区域占位置

如果不治理，结果通常是：

- 开发时不知道该以哪份文档为准
- 重构完成后，下一步方向仍然不清楚
- 文档越多，反而越难推动项目

所以文档管理的目标不是“更全”，而是“更能驱动决策和执行”。

## 文档飞轮的六个环节

### 1. 产品目标文档

作用：

- 定义下一阶段到底要赢什么
- 防止工程工作失去产品锚点

当前对应文档：

- `session-run-closure-requirements.md`

要求：

- 只能有一份真正的主目标文档
- 其他文档要么是阶段执行包，要么是历史决策记录
- 一旦新目标成立，旧目标文档应降级

### 2. 领域边界文档

作用：

- 限制系统不要偷偷长出新的对象和错误抽象
- 保证产品演进仍然和 session-first 架构一致

当前对应文档：

- `core-domain-contract.md`
- `session-first-workflow-surfaces.md`
- `product-surface-lifecycle.md`

要求：

- 这层文档必须稳定
- 只有当产品边界真的变了时才更新
- 不能让实现细节反向污染领域契约

### 3. 架构与拆分文档

作用：

- 把产品目标映射成代码层面的结构调整
- 帮助重构不偏离产品方向

当前对应文档：

- `project-architecture.md`
- `core-domain-implementation-mapping.md`
- `core-domain-refactor-todo.md`
- `structural-cleanup-plan.md`

要求：

- 每次重构都要能落回到这层文档
- 文档必须回答“该拆哪里、为什么拆、拆完对主流程有什么帮助”

### 4. 功能与专项方案文档

作用：

- 承接某一块独立能力的落地设计和部署方式

当前对应文档：

- `hooks-and-node-structure.md`
- `object-storage-file-assets.md`
- `external-message-protocol.md`
- `voice-connector.md`
- 以及各类 connector / integration 文档

要求：

- 只保留仍然会被使用的专项文档
- 专项文档不能替代产品主线文档
- 与主线脱节的专项方案应降级或归档

### 5. 测试与验收文档

作用：

- 把“我们以为完成了”变成“有验收依据地完成了”

当前项目里这层还比较弱，后续应加强。

理想形态：

- 每条主线需求文档都能指向对应测试
- 每次重要重构都能说明要守住哪些回归测试
- 每个主要产品目标都要有自己的 Definition of Done

### 6. 复盘与清理文档

作用：

- 把阶段性结论回收进主线
- 把失效文档降级或删除

当前对应文档：

- `refactor-opportunities.md`
- `session-state-audit.md`
- 各类阶段性 note

要求：

- 不是所有探索都应永远留在 `current`
- 一旦结论已经进入主线文档，阶段性 note 应考虑归档

## 当前最合理的文档分层

基于当前项目状态，建议把文档理解为四层。

### 第一层：当前真相

回答“系统现在是什么”。

包括：

- `docs/current-features.md`
- `docs/project-architecture.md`
- `notes/current/core-domain-contract.md`
- `notes/current/core-domain-implementation-mapping.md`

这是最重要的一层，任何实现和重构都必须回到这里校对。

### 第二层：当前主线

回答“接下来往哪走”。

包括：

- `notes/current/session-run-closure-requirements.md`
- `notes/current/core-domain-refactor-todo.md`
- `notes/current/session-first-workflow-surfaces.md`
- `notes/current/product-surface-lifecycle.md`

这一层要保持聚焦，不能让过多旧阶段文档挤占主线。

### 第三层：工程支撑

回答“怎么更稳、更快、更容易维护”。

包括：

- `docs/structural-cleanup-plan.md`
- `docs/refactor-opportunities.md`
- `notes/current/performance-optimization-checklist.md`
- `notes/current/self-hosting-dev-restarts.md`

### 第四层：专项能力

回答“某个独立能力怎么接入、部署和运行”。

包括：

- connector / integration 文档
- 文件资产、语音、GitHub、Feishu 等专题文档

## 每层文档的管理动作

为了让你真正有掌控感，文档不能只有分类，还要有明确管理动作。

### 当前真相

管理动作：

- 必须保持最新
- 每次影响产品面或架构真相的改动都应同步更新
- 不允许长期与代码现实脱节

### 当前主线

管理动作：

- 只保留少量高优先级文档
- 同一阶段只能有一份主目标总纲
- 每次阶段目标变化时优先更新这一层

### 工程支撑

管理动作：

- 按需维护
- 在重构、性能、稳定性工作中引用
- 不应反客为主变成产品路线文档

### 专项能力

管理动作：

- 只在对应能力仍存在且仍被维护时保留
- 进入低活跃状态后应考虑归档
- 不应挤占主线入口

## 文档如何产生项目飞轮

文档要产生飞轮，关键不是写得多，而是让每一层都向下一层施加约束，并从下一层收回结果。

### 飞轮链路

#### 产品目标 -> 领域边界

产品目标提出后，必须先检查是否突破当前领域契约。

例如：

- “任务地图成为主舞台”可以做
- “顺手引入一个 durable Task object”不可以直接做

这一步避免产品讨论一开始就跑偏。

#### 领域边界 -> 架构拆分

当领域边界明确后，才知道该怎么拆代码。

例如：

- 如果 session 仍是 canonical object，那么 workbench 和 node 就必须是派生层
- hooks 是编排机制，不应变成另一套工作流 authority

这一步避免重构时抽象错位。

#### 架构拆分 -> 功能实现

当代码边界清楚后，功能开发才不会继续堆在单个大文件里。

例如：

- session lifecycle hooks 应从 session-manager 中抽出
- task-map projection 应保持在派生层，不反向写成新的底层对象

这一步让实现更容易持续推进。

#### 功能实现 -> 测试验收

任何真正进入主线的功能都要有回归依据。

例如：

- hooks 注册与触发测试
- task-map node 生成测试
- 任务闭环状态机验收测试

这一步让“文档里的设想”真正变成“项目里的能力”。

#### 测试验收 -> 复盘清理

功能稳定后，应回收和清理阶段性文档。

例如：

- 临时执行包可以降级
- 早期讨论稿可以 archive
- 主线文档可以更新成新的当前真相

这一步防止文档堆积成噪音。

#### 复盘清理 -> 下一轮产品目标

当文档和代码都被清理后，下一轮产品目标才有干净起点。

这就是飞轮闭环。

## 当前最需要补强的地方

基于现状，文档体系里最需要补强的是这三点：

### 1. 主目标文档只能有一个

当前主流程类文档存在重叠：

- `session-run-closure-requirements.md`

后续应明确：

- 哪一份是当前唯一主目标
- 哪些是历史决策记录
- 哪些是执行包

### 2. 需求文档必须绑定实现和测试

当前很多文档定义了方向，但还没有形成“文档 -> 文件 -> 测试”的固定链路。

后续每份主线文档最好补三样东西：

- 相关模块
- 关键接口
- 验收测试

### 3. current 区需要持续降噪

`notes/current/` 只能放“现在还在生效”的内容。

一旦文档只剩历史价值，就应该：

- 合并入主文档
- 迁入 archive
- 或直接删除

## 现在就可以执行的文档治理动作

结合当前项目状态，建议按下面方式管理：

1. 把 `session-run-closure-requirements.md` 作为后续产品方向总纲
2. 把 `core-domain-contract.md` 和 `session-first-workflow-surfaces.md` 作为边界约束，不轻易改动
3. 把 `core-domain-refactor-todo.md` 作为当前重构执行面，随着重构推进及时回写
4. 阶段性执行包应及时归档或删除，不再和主线文档并列竞争
5. 把专项方案文档只保留在对应专题入口，不作为日常主线阅读材料

## 建议的文档治理规则

后续可按下面规则管理：

### 规则 1：新增文档前先问四个问题

1. 这是当前真相，还是阶段讨论？
2. 它属于产品目标、领域边界、架构拆分、专项方案中的哪一层？
3. 它会在两周后仍然有效吗？
4. 它是否已经能映射到代码或测试？

如果回答不清楚，就不应该新建主线文档。

### 规则 2：每个主线阶段只保留一份总纲

例如当前后续方向，应以：

- `session-run-closure-requirements.md`

作为任务闭环方向总纲。

其他文档应作为：

- 决策记录
- 过渡执行包
- 历史背景

而不是继续并列竞争。

### 规则 3：文档更新必须伴随代码或路线变化

不要把文档当成单独世界。

应该形成：

- 路线变了 -> 更新需求文档
- 架构变了 -> 更新 architecture / mapping
- 功能落了 -> 更新 current features 和测试引用

### 规则 4：定期做文档回收

每完成一轮较大迭代后，应做一次：

- 主线文档校正
- 重叠文档降级
- 阶段文档归档
- 索引更新

## 对当前项目最有价值的结果

如果按这套方式管理，文档体系对项目真正产生的飞轮会是：

1. 产品方向更少摇摆
2. 重构更能对齐主目标
3. 功能开发更容易落到正确层级
4. 测试更能跟上需求而不是只跟着实现细节
5. 文档不会越来越多，而会越来越聚焦

最终目标不是“文档齐全”，而是：

**文档能持续帮助 MelodySync 做出更快、更准、更可回收的产品与工程决策。**
