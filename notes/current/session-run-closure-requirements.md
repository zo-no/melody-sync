# 一次任务闭环需求文档

状态：draft

目的：定义 MelodySync 在后续重构完成后要继续推进的顶层产品目标，即让用户从第一条消息开始，在同一个工作区内完成一次任务的发起、推进、分支、收束、完成判断与完成后评估，形成完整闭环。

## 一句话目标

让用户从发送第一条消息开始，就进入一次任务生命周期；之后所有任务推进、支线管理、下一步推荐、完成判断和完成后分析，都在同一个工作区内完成，而不是分散在会话列表和零散状态提示中。

## 目标效果

用户使用产品时，应感受到这不是“聊一段对话然后自己整理”的工具，而是“在一个工作区里完成一件事”的工具。

理想体验如下：

1. 用户发送第一条消息，任务生命周期启动
2. 系统逐步识别目标、推进主线、发现支线、推荐下一步
3. 当任务出现分叉时，系统在任务地图上给出支线建议，而不是把提示散落在别的界面
4. 当部分工作完成时，系统能把支线结论回带主线
5. 当任务接近完成时，系统能判断是否已经收束
6. 当任务完成时，系统继续做结果分析和评估，而不是停在“回答结束”
7. 用户始终在同一个工作区里理解任务状态、继续任务、结束任务

## 核心设计结论

### 1. 任务地图将成为任务流程管理的主舞台

后续产品方向中，任务地图不再只是 workbench 的一个辅助视图，而应逐步成为任务生命周期的主操作面。

这意味着：

- 支线推荐应进入任务地图
- 下一步推荐应进入任务地图
- 任务完成判断应进入任务地图
- 完成后分析和评估也应进入任务地图

任务流程管理不应继续分散在会话列表、局部提示、零散字段中。

### 2. 会话列表退回到“记录和索引层”

会话列表应主要承担以下作用：

- 历史记录
- 快速索引
- 最近活跃信息
- 名称、分组、归档等基础组织能力

会话列表不再承担任务流程管理职责，尤其不应继续承担：

- 支线任务推荐入口
- 下一步问题推荐入口
- 任务闭环判断入口

这些能力都应逐步迁移到任务地图所在的工作区中。

### 3. hooks 是闭环引擎，node 是闭环表达

在这条闭环链路里：

- `hooks` 负责生命周期中的状态管理、自动编排、后处理
- `node` 负责把当前状态和下一步动作表达在任务地图中

两者关系不是平级竞争，而是：

```text
任务生命周期事件
  -> hooks 调度和状态编排
  -> workbench / 任务态整理
  -> 任务地图 node 表达
```

## 术语定义

### Session

`Session` 仍然是当前架构里的 canonical durable object。

它承载：

- 对话历史
- 任务相关结构化状态
- 运行时偏好
- 分支 lineage

当前架构下，这一点不变。

### Workbench

`Workbench` 是基于 session 派生出的任务组织层，不是新的 durable object。

它的作用是把原本分散在会话里的任务信息整理成“任务视角”，帮助用户判断：

- 当前目标是什么
- 现在任务进行到哪里
- 有哪些支线
- 下一步该做什么
- 是否已经收束

### 任务地图

任务地图是 workbench 中最核心的一种图形化展示层。

它用节点和关系来表达：

- 主线
- 目标
- 支线
- 候选支线
- 下一步
- 收束状态
- 完成后评估

本轮需求中，任务地图将逐步成为 workbench 的主操作面。

### Hooks

这里的 `hooks` 不是 React hooks，而是 run / session 生命周期中的回调调度机制。

它负责：

- 在任务不同阶段执行自动状态管理
- 在 run 结束后做 post-run reconciliation
- 推动任务节点状态变化
- 生成支线建议、下一步建议、完成判断结果、完成后分析等中间产物

Hooks 是任务生命周期的编排器，不是任务真相本身。

### Node

这里的 `node` 不是 Node.js，而是任务地图中的节点语义。

Node 不是底层持久化对象，而是任务状态与动作建议的可视化表达单元。

## 顶层产品目标

我们要做的不是“让一次 run 完成”，而是“让一次任务完成闭环”。

因此必须区分三层概念：

- `run`：一次执行
- `session`：一个 durable work thread
- `task lifecycle`：从第一条消息开始到任务真正收束的完整过程

这意味着：

- 一次任务生命周期不等于一次 run
- 一次任务生命周期也不一定只等于一次简单问答
- run 只是任务推进过程中的一个动作单元
- session 是任务生命周期的承载体
- hooks 是生命周期编排层
- 任务地图是生命周期展示和操作层

## 想要达成的任务闭环

从产品角度，一次任务闭环至少应包含这些阶段：

1. 任务发起
2. 目标识别
3. 主线推进
4. 支线发现或推荐
5. 支线执行与回带
6. 下一步推荐
7. 完成判断
8. 完成后分析与评估
9. 任务收束

当前重构完成后，后续迭代应围绕这 9 个阶段推进。

## hooks 在闭环中的职责

hooks 不再只意味着“run 完成后发个通知”。

在这套设计里，hooks 的职责应升级为任务生命周期状态编排，包括但不限于：

1. 在任务开始时建立初始任务态
2. 在 run 完成后更新任务状态
3. 在 run 完成后判断是否出现新的支线推荐
4. 在 run 完成后生成下一步建议
5. 在 run 完成后判断任务是否已完成
6. 在任务完成后触发分析和评估
7. 在目标已知的前提下，持续维护整条任务主线的推进状态

因此，hooks 更准确的角色应是：

**任务生命周期状态编排器**

## node 在闭环中的职责

如果任务地图成为主操作面，那么 node 也不再只是静态展示组件，而是任务管理语义本身。

当前已明确的 node 职责包括：

1. 支线任务节点推荐
2. 下一步问什么推荐
3. 判断任务是否完成
4. 任务完成后进行分析和评估
5. 在已知目标的情况下，管理整条任务流程

这意味着 node 未来至少要覆盖两大类：

### 状态节点

表示任务当前状态，例如：

- `main`
- `branch`
- `done`
- `reviewed` / `evaluated`

### 动作节点

表示系统建议用户下一步做什么，例如：

- `candidate`
- `next-step`
- `ask-next`
- `investigate`
- `merge-back`

这个区分很重要，因为只有把“状态”与“动作”分开，任务地图才既能说明现在处于什么状态，又能明确告诉用户下一步怎么走。

## 推荐的任务生命周期主链路

重构后继续做时，建议按以下主链路设计闭环：

### 1. 用户发送第一条消息

系统应：

- 建立主任务线
- 初始化任务生命周期
- 如果能识别主任务语义，则更新 taskCard 的 goal / mainGoal 字段，但不额外生成 goal node

### 2. run 推进

系统应：

- 持续沉淀任务状态
- 识别当前是在主线推进还是支线推进

### 3. run 结束

系统应通过 hooks 判断：

- 是否出现新的 branch candidates
- 是否应该生成 next-step recommendation
- 当前任务是否已经达到可完成状态

### 4. 若任务未完成

系统应在任务地图中显示：

- 候选支线
- 下一步推荐问题或动作

### 5. 若任务部分完成

系统应在任务地图中显示：

- 已完成支线
- 可回带主线的结论

### 6. 若任务完成

系统应：

- 生成完成判断结果
- 生成 done / closure 信号
- 触发完成后分析与评估 hooks
- 在任务地图中呈现 review / evaluation 结果

### 7. 用户继续在任务地图中推进或结束任务

用户后续的主要操作入口应仍然在工作区中，而不是退回会话列表重新寻找流程提示。

## 当前这轮闭环需要回答的关键问题

为了真正落地，后续继续做时必须先回答下面 3 个问题：

### 1. 一次任务生命周期的最小状态机是什么

必须先明确：

- 从开始到结束最少有哪些状态
- 哪些状态之间允许跳转
- 哪些状态意味着任务还在推进
- 哪些状态意味着任务已经收束

如果没有状态机，hooks 触发点和 node 类型都会漂移。

### 2. 第一版任务地图最少需要哪些 node

必须先收敛：

- 哪些 node 是最小可用集合
- 哪些 node 是后续扩展
- 哪些 node 是状态节点
- 哪些 node 是动作节点

### 3. hooks 在哪些生命周期时机触发，并产出什么

必须先定义：

- 哪些事件触发 hooks
- 每类 hook 会更新哪些状态
- 每类 hook 给任务地图产出哪些 node 素材

## 当前建议的最小能力集合

如果后续按“先做一版真正闭环”推进，建议最小集合如下。

### hooks 侧最小集合

- 任务开始时初始化任务态
- run 完成后更新任务状态
- 支线候选生成
- 下一步推荐生成
- 完成判断
- 完成后分析和评估

### node 侧最小集合

- `main`
- `candidate`
- `branch`
- `next-step`
- `done`
- `evaluation`

## 非目标与约束

### 不引入新的 Task object

本轮仍然坚持 session-first，不引入新的 durable `Task` 对象。

### 不让会话列表重新承担任务管理

会话列表可以保留索引和记录职责，但不应继续承担任务流程组织职责。

### 不让 hooks 成为隐藏 domain model

hooks 可以编排状态变化，但不应自己成为任务领域真相。

### 不让 node 成为被持久化的 workflow authority

node 必须始终是派生表达，而不是需要单独持久化的一层真实对象。

## 验收标准

后续重构完成并继续推进时，如果要判断这条产品方向是否成立，至少应满足：

1. 用户从第一条消息开始，就能进入一次清晰的任务生命周期
2. 任务推进、分支、下一步推荐和完成判断都主要发生在工作区内
3. 会话列表只承担记录和索引，不再承担任务流程推荐
4. hooks 能在任务不同阶段完成状态管理和自动编排
5. 任务地图能清晰表达当前状态和下一步动作
6. 任务完成后系统不会停在“回答结束”，而会继续做完成后分析与评估
7. 用户最终可以在一个工作区内完成一次任务的完整闭环

## 结语

这份文档定义的不是一个局部功能点，而是一条顶层产品路线：

**让任务地图成为任务生命周期主舞台，让 hooks 驱动状态管理与流程推进，让用户在一个工作区内完成一次任务的完整闭环。**

后续在项目重构完成后，应从“任务生命周期状态机”“最小 node 集合”“hooks 触发与产出契约”这三件事继续往下拆。
