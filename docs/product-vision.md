# MelodySync 产品愿景文档

> 核心目标：让用户在使用过程中达到注意力高度集中，进入心流状态。
> 终极目标：人类逐步把可重复的事情交给 AI，把精力集中在真正需要人类判断的事情上。

---

## 一、两个 Tab 的本质区别

### 1.1 日常任务 Tab = 任务类型维度

按**任务的性质**分类，不管任务归属于哪个项目。

```
日常任务 Tab
├── [顶部总控制面板]  ← 全局任务健康度、今日完成情况
├── 长期任务          ← 所有项目里 bucket=long_term 的任务
├── 短期任务          ← 所有项目里 bucket=short_term 的任务
├── 等待任务          ← 所有项目里 bucket=waiting 的任务
├── 收集箱            ← 所有项目里 bucket=inbox + 无归属的任务
└── 快捷按钮          ← 常用/全部，由眼睛按钮控制
```

**眼睛按钮的语义：**
- 眼睛**打开** = 显示所有任务（含已归属长期项目的）
- 眼睛**关闭** = 只显示**未归属任何长期项目**的任务（纯临时任务）
- 快捷按钮部分：眼睛打开 = 全部快捷按钮；眼睛关闭 = 只显示"已标记为常用"的快捷按钮

### 1.2 长期项目 Tab = 项目归属维度

按**任务归属于哪个项目**分类，看的是项目结构和进展。

```
长期项目 Tab
├── 日常任务（默认系统项目）
├── MelodySync 系统管理
├── 理财流水线
└── 摄影计划
```

### 1.3 两个维度的关系

同一个任务可以在两个 Tab 里都看到，只是呈现方式不同：
- 日常任务 Tab：按类型聚合，看"今天要做哪类事"
- 长期项目 Tab：按项目聚合，看"这个项目进展如何"

## 二、顶部总控制面板

放在日常任务 Tab 最顶部，类似长期项目的控制面板，但管理全局。

**展示内容：**
- 今日完成情况（正反馈，从 worklog 读取）
- 各类型任务数量：长期 N / 短期 N / 等待 N / 收集箱 N
- 各项目活跃度统计
- 行为模式推荐（"你最近经常做饭，要不要创建厨艺提升项目？"）

## 三、快捷按钮（Button）

### 3.1 概念定义

**快捷按钮** ≠ **Skill（Agent 能力）**

| 维度 | 快捷按钮（Button） | Skill（Agent 能力）|
|------|-------------------|-------------------|
| 是什么 | 用户在 MelodySync 里一键触发的功能 | Agent 能力库里的具体执行单元 |
| 从哪里来 | 从对话/任务中"长出来" | 在 Agent 系统里定义 |
| 包含什么 | AI 指令 + 可执行代码 | 纯 Agent 能力 |
| 未来关系 | 快捷按钮里可以调用指定的 Skill | 被快捷按钮调用 |

### 3.2 快捷按钮的能力

快捷按钮可以包含：
1. **AI 指令**：告诉 AI 做什么（runPrompt）
2. **可执行代码**：Shell 脚本、API 调用等
3. **Skill 调用**：调用 Agent 系统里的某个 Skill
4. **检查点**：需要人类确认才继续执行的节点

### 3.3 内置常用快捷按钮（建议）

| 按钮名 | 功能 | 是否需要检查点 |
|--------|------|---------------|
| 整理任务 | AI 扫描所有任务，归档完成的，整理收集箱 | ✓ 确认删除前 |
| 今日回顾 | 回顾今天的对话，生成日记摘要 | ✗ |
| 每日清理 | 归档今日所有任务，写柳比歇夫日记 | ✗ |
| 拉取最新代码 | 执行 git pull，检查状态 | ✗ |
| 项目周报 | 生成当前项目的周报摘要 | ✗ |
| 深度回顾 | 分析近期任务模式，提出改进建议 | ✓ 执行前确认 |

### 3.4 从任务到快捷按钮的转化路径

**用户路径：做着做着发现这是个按钮**

```
1. 用户在做一个短期任务（例：整理任务列表）
2. 做完后，AI 在对话中生成了清晰的步骤
3. 用户觉得"这个我以后还会用"
4. 点击 workbench 底部 "设为快捷按钮"
   → 弹出配置面板：
     - 按钮名称（默认用任务名）
     - 触发指令（从当前对话 runPrompt 提取）
     - 是否包含代码（可选）
     - 是否需要检查点（可选）
     - 标记为"常用"（影响眼睛关闭时是否显示）
5. 保存后，按钮出现在日常任务 Tab 的快捷按钮区域
```

### 3.5 快捷按钮 UI 设计

**视觉风格：真正的按钮感**
- 不是普通的列表项，而是卡片式按钮
- 图标 + 名称 + 简短描述
- 悬停时有按下感（transform scale）
- 运行中显示加载动画
- 有检查点的按钮显示特殊标识（⚠️ 或 🔒）

**布局：**
```
快捷按钮区域
┌─────────────┐ ┌─────────────┐ ┌─────────────┐
│  📋 整理任务  │ │  📓 今日回顾  │ │  🔄 每日清理  │
│  归档完成任务  │ │  生成日记摘要  │ │  写柳比歇夫   │
└─────────────┘ └─────────────┘ └─────────────┘
```

### 4.1 任务地图结构

**日常任务视图（左侧边栏 "日常任务" Tab）**

```
日常任务 Tab
├── [总控制面板]
├── 长期任务（来自所有项目）
├── 短期任务（来自所有项目）

└── 快捷按钮（卡片式，常用/全部）
```

**单一长期项目视图（进入项目后）**

```
绘画学习（根节点）
├── 长期任务（AI 自动循环执行）
│   ├── 每日读图
│   └── 每周美术史
├── 短期任务（定时执行一次）
├── 等待任务（等待人类处理）
├── 收集箱（待整理的想法）
└── 快捷按钮（该项目专属的常驻功能）
```

### 4.2 任务状态流转路径

```
随手记录
  ↓
收集箱（无压力积累）
  ↓ AI 整理建议
长期/短期/等待任务（明确类型）
  ↓ 发现可以持续做
归入长期项目（明确归属）
  ↓ 建立循环
AI 自动执行（长期/短期任务）
  ↓ 发现是常驻功能
快捷按钮（一键触发，可含代码+AI+Skill）
  ↓ 人类彻底交权
AI 全自动（人类只看结果）
```

## 五、心流保护机制

### 5.1 进入项目 = 进入专注模式

用户点击长期项目 → 界面只展示这个项目的内容：
- 左侧边栏：只显示该项目的任务
- 右侧任务地图：只显示该项目的节点
- 其他项目完全隐藏，不产生干扰

### 5.2 日常任务 = 全局视角

日常任务 Tab 是唯一能看到所有任务的地方：
- 按任务类型聚合，不区分项目归属
- 眼睛关闭 = 只看未归属的临时任务（最纯粹的今日清单）

## 六、每日清理与数据沉淀

### 6.1 每日清理（默认长期任务）

**触发时间**：每天零点（可配置）

**清理逻辑**：
1. 归档日常任务中所有今日任务（完成/未完成都归档）
2. 统计完成率，写柳比歇夫日记
3. 生成正反馈摘要（"今日完成 X 项，完成率 Y%"）
4. 更新 context-digest.md
5. 分析近 30 天行为模式，写入 Behavior Patterns

## 七、AI 介入机制

### 7.1 被动介入：任务完成后的分类建议

用户完成一个任务后，AI 判断是否值得持续做、归入项目、创建项目。

### 7.2 主动介入：从日常任务创建长期项目

用户说"我想学绘画" → AI 识别持续性意图 → 提议创建长期项目 → 帮助拆解子任务结构。

### 7.3 智能推荐：从行为模式提炼长期项目

每日清理时分析 30 天模式，发现重复主题 → 推荐创建长期项目。

### 7.4 快捷按钮的检查点机制

当快捷按钮包含**破坏性操作**（删除、归档大量数据等）时：
1. AI 执行到检查点暂停
2. 在对话中展示"即将执行的操作清单"
3. 用户确认 → 继续执行
4. 用户拒绝 → 取消或修改

## 八、人类作为 Tool/Skill

### 8.1 核心设想

随着长期项目积累，AI 对用户能力有清晰认知，可以把人类能力抽象成"技能"，让 AI Agent 在发出等待任务时主动调用。

### 8.2 技能积累路径

```
长期项目执行 → 产出结论（knownConclusions）
  → 每日清理归纳 → 技能摘要
  → 写入 memory/skills.md
  → AI 发出等待任务时可引用
```

## 十、任务类型扩展：审批与信息传递

### 10.1 当前任务类型

| 类型 | 执行者 | 触发方式 | 说明 |
|------|--------|----------|------|
| 长期任务（recurring_task） | AI | 周期性自动 | AI 循环执行的长期任务 |
| 短期任务（scheduled_task） | AI | 定时执行一次 | AI 在固定时间点执行 |
| 等待任务（waiting_task） | 人类 | 手动触发 | 等待人类处理，AI 准备好后续步骤 |
| 快捷按钮（skill） | AI/混合 | 一键触发 | 人类点击触发，AI 执行 |

### 10.2 扩展：等待任务的子类型

等待任务本质上是"人类要做的任务"，可以细分为：

**审批任务（approval_task）**
- AI 准备好操作清单，等待人类确认
- 典型场景：批量删除、归档大量数据、执行破坏性操作
- 交互：展示"待审批面板"，用户逐项确认或全部批准
- 确认后 AI 继续执行

**信息传递任务（input_task）**
- AI 需要人类提供信息才能继续
- 典型场景：填写资金记录、提供决策依据、输入账号密码
- 交互：展示表单或提示，用户填写后 AI 继续

**当前实现**：两者都用 `waiting_task`，通过 `runPrompt` 描述具体需要什么。
**未来扩展**：可以在 `persistent.kind` 增加 `approval_task` 和 `input_task` 子类型，或通过 `persistent.waitingType` 字段区分。

### 10.3 审批模式（Approval Mode）

**触发时机**：
- AI 准备执行破坏性操作（删除、归档大量任务）
- 快捷按钮包含检查点（`hasCheckpoint: true`）
- 用户明确要求"需要我确认"

**交互流程**：
```
AI 执行到检查点
  ↓
创建 waiting_task（waitingType: "approval"）
  ↓
侧边栏展示审批面板（站内通知）
  ↓
用户查看操作清单，逐项确认/拒绝
  ↓
用户点击"批准执行"
  ↓
AI 继续执行已批准的操作
```

**审批面板 UI**：
- 出现在等待任务的 bucket 里，有特殊的 🔐 标识
- 展示"待执行操作清单"
- 支持全部批准 / 全部拒绝 / 逐项选择
- 批准后自动触发 `run-persistent`

### 10.4 实现路径

**短期（P2）**：
- 在 `waiting_task` 的 `runPrompt` 里描述审批内容
- AI 执行前创建 waiting_task 列出操作清单
- 用户手动触发后 AI 继续

**中期（P3）**：
- 在 persistent 里加 `waitingType: "approval" | "input"` 字段
- 审批任务在 UI 上有专属样式（🔐 图标、橙色边框）
- 支持在对话里直接点击批准按钮

### P4（审批与信息传递）
- [ ] `waitingType` 字段：区分 approval / input / general
- [ ] 审批面板 UI：展示操作清单，支持逐项确认
- [ ] 快捷按钮检查点：执行到检查点时自动创建审批任务
- [ ] AI 操作规则：涉及删除/归档时必须先创建审批任务

## 十二、信息透出原则（Information Disclosure）

> 每一个显示在界面上的信息，都必须有明确的用户价值。看得见的信息不用文字解释，用户自己会看。

### 12.1 原则

**只透出用户此刻需要采取行动或做出判断的信息。**

- 信息已经通过 UI 结构表达的，不再用文字重复
- 详情放在用户主动进入的地方（点击进控制面板/详情页），不主动推送
- 状态 chip 足够表达归属和状态，不需要额外说明文字

### 12.2 Workbench 顶部信息区（trackerPersistentSummary / trackerBranch）

| 信息 | 是否透出 | 原因 |
|------|---------|------|
| 任务归属（`归属 MelodySync 系统管理` chip）| ✅ | 用户需要知道当前会话属于哪个项目 |
| 任务状态（`自动执行中` / `等待中` chip）| ✅ | 用户需要知道 AI 当前在做什么 |
| 下次触发时间（`下次 明天 09:00` chip）| ✅ | 用户需要知道何时会自动触发 |
| checkpoint/summary 文字（「确认先一文为核心...」）| ❌ | 用户打开会话就能看到对话内容，重复显示无意义 |
| 长期任务归属的解释文字（「左侧列表继续保留...」）| ❌ | UI 结构本身已经说明，文字是噪音 |
| 建议归入的解释文字（「当前这条会话更像属于...」）| ❌ | chip 已经表达，文字多余 |
| 项目摘要（「产品自维护：功能迭代...」）| ❌ | 用户想看摘要会点进控制面板 |

### 12.3 项目卡片（Projects Tab）

| 信息 | 是否透出 | 原因 |
|------|---------|------|
| 项目名称 | ✅ | 识别项目 |
| 任务数量 | ✅ | 一眼知道项目规模 |
| 项目摘要/描述 | ❌ | 用户点进控制面板自然看到，卡片里重复是噪音 |
| 「控制面板 ›」入口 | ✅ | 明确告知可以进入，但不展示内容 |

### 12.4 任务列表条目

| 信息 | 是否透出 | 原因 |
|------|---------|------|
| 任务名称 | ✅ | 识别任务 |
| 归属项目 tag | ✅ | 知道属于哪个项目 |
| 状态（进行中/等待/完成）| ✅ | 知道当前状态需不需要介入 |
| checkpoint 预览文字 | ✅（仅当前活跃任务）| 当前任务的进展是用户最关心的 |
| 已知结论展开详情 | ❌ | 用户点进会话自然看到，不需要在列表里展开 |

### 12.5 任务列表交互

| 交互 | 期望行为 | 原因 |
|------|---------|------|
| 点击任务条目 | 切换到该会话（右侧内容切换），左侧高亮跟随 | 选中即切换，状态一致 |
| 点击任务条目 | 不展开详情面板 | 详情是主动查看行为，不是点击默认结果 |
| hover 任务条目 | 显示操作按钮（归档/删除/置顶/重命名）| 操作在需要时出现，不占空间 |

### 12.6 任务状态与操作（TodoList 风格）

| 状态 | 显示位置 | 操作按钮 | 用户意图 |
|------|---------|---------|---------|
| 活跃 | 任务列表主体 | ○（完成圆圈）· ✏️ 重命名 · 📌 置顶 · 🗑 删除 | 正在进行中 |
| 完成（✓）| 底部「已完成」分组，UI 变灰 | ✓（撤销完成圆圈）· ✏️ 重命名 · 🗑 删除 | 已处理完毕 |

**设计原则：**
- 完成 ≠ 归档：完成只是 UI 状态变化（移到底部、变灰），任务仍然可见
- 「归档」概念从整个项目移除，后端数据模型用 `workflowState: "done"` 表示完成
- **删除**是每个任务都有的操作，直接永久删除，弹一次确认框，无需前置条件
- 不需要「先归档再删除」的两步流程
- 「已完成」分组可折叠，默认展开
- 完成圆圈（○/✓）是核心交互，所有普通任务都有；AI 自动任务（recurring/scheduled）和快捷按钮除外

**数据模型：**
- 完成状态：`workflowState: "done"`（原 `archived: true` 废弃）
- 撤销完成：`workflowState: ""`
- 删除：直接 DELETE，无前置条件
